# mqConnector

A production-grade message-queue bridge and pipeline runner. Consumes from one of {IBM MQ, RabbitMQ, Kafka} and forwards to another, optionally applying validation, field filtering, structural transforms, format translation (XML ↔ JSON), content-based routing, and scriptable transforms.

Built as a single Go binary with an embedded SvelteKit admin UI, structured logging, SQLite-backed configuration, SimpleAuth, TLS by default, and Prometheus metrics.

## Capabilities

- **Connectors:** IBM MQ (build-tag `ibmmq`, CGO), RabbitMQ (AMQP 0.9.1), Kafka (sarama)
- **Pipeline stages:** validate → filter → transform → translate → route → script
- **Storage:** SQLite (modernc, pure Go — no CGO required for the default build)
- **Auth:** SimpleAuth (no third-party identity provider; works in air-gapped deployments)
- **Hot reload:** pipeline definitions reload on update, no restart required
- **DLQ:** failed messages persisted with retry counters; retry endpoint actually re-publishes
- **Metrics:** Prometheus exposition at `/api/metrics/prometheus`
- **TLS:** mandatory in non-dev modes (self-signed acceptable for dev)
- **UI:** SvelteKit + Tailwind, bilingual (en/ar) with full RTL, dark + light themes, brand-compliant
- **Single binary:** the SvelteKit static build is embedded via `go:embed`

## Build

```sh
# Default build — no CGO, no IBM MQ, RabbitMQ + Kafka only
./scripts/build.sh

# Native build with IBM MQ on a linux/amd64 host (requires CGO + ibmmq_dist/)
./scripts/build.sh --ibmmq

# Cross-platform IBM MQ build via Docker (recommended on macOS/ARM)
docker buildx build --platform linux/amd64 \
  --build-context simpleauth-sdk=../SimpleAuth/sdk/go \
  -f Dockerfile.ibmmq -t mqconnector:ibmmq .
```

The script:
1. Builds the SvelteKit frontend with the static adapter into `internal/web/dist/`
2. Compiles the Go binary into `dist/mqconnector`
3. Stamps the version from `VERSION` into the binary

The OpenAPI 3.0 spec lives at `internal/server/openapi.yaml` and is
served live at `/api/openapi.yaml`. SDK generators and external doc
tooling can fetch it directly from any running instance.

## Run (native)

```sh
cp config.example.yaml config.yaml
./scripts/dev.sh         # development (TLS optional, verbose logging)
./dist/mqconnector       # production (TLS required, JSON structured logs)
```

Default admin URL: `https://localhost:8443/`

## Run (Docker)

A `docker-compose.yml` brings up the full local stack — mqConnector, SimpleAuth (built from the sibling project), RabbitMQ, Kafka in KRaft mode, and a one-shot self-signed TLS cert generator:

```sh
cp .env.example .env
# Tweak passwords and ports in .env (notably SIMPLEAUTH_ADMIN_PASSWORD)
docker compose up -d
```

Service URLs:

| Service | URL / port |
|---|---|
| mqConnector admin UI | `https://localhost:8443/` |
| SimpleAuth | `https://localhost:9443/` |
| RabbitMQ management | `http://localhost:15672/` |
| RabbitMQ AMQP | `amqp://localhost:5672/` |
| Kafka bootstrap | `localhost:9092` |

Notes:
- The compose build uses BuildKit's `additional_contexts` to bring the sibling `SimpleAuth/sdk/go` into the mqConnector image (the SDK is consumed via a `replace` directive in `go.mod`). If your SimpleAuth checkout is elsewhere, set `SIMPLEAUTH_CONTEXT` and `SIMPLEAUTH_SDK_PATH` in `.env`.
- The default cert is self-signed and generated by the `certs-init` service into a named volume. For production, mount real TLS material under `/etc/mqconnector/tls/` and set `MQC_MODE=prod`.
- Healthchecks are wired on every service; `docker compose ps` will show the stack as healthy once SimpleAuth is reachable and mqConnector has bound its listener.
- Standalone `docker build` (without compose) requires the named context too: `docker build --build-context simpleauth-sdk=../SimpleAuth/sdk/go -t mqconnector .`

## Configuration

All configuration is in `config.yaml`. Every value can be overridden by an environment variable (uppercase, dot replaced by underscore — e.g. `server.tls.enabled` → `SERVER_TLS_ENABLED`). See `config.example.yaml` for the full reference.

## Tests

```sh
go test ./...                  # full suite, ~9 s
go test -race ./...            # race-detected suite, ~10 s
go test -bench=. ./...         # per-stage + pool benchmarks
go test -cover ./...           # per-package coverage summary
```

End-to-end scenario coverage (no external broker required) lives in
[`internal/pipeline/e2e_test.go`](internal/pipeline/e2e_test.go): it
seeds storage, uploads a JSON Schema, configures filter + transform
stages, drives messages through in-memory MQ connectors, and asserts
both the destination payload and the DLQ contents.

A real-broker integration test
([`internal/pipeline/integration_rabbit_test.go`](internal/pipeline/integration_rabbit_test.go))
publishes through a live RabbitMQ to catch wire-format regressions the
in-memory connector can't see. It's behind the `integration` build tag
and skipped without `RABBIT_URL`:

```sh
docker compose up -d rabbitmq
RABBIT_URL=amqp://mqc:mqc-dev@localhost:5672 \
  go test -tags integration -run TestIntegration_RabbitMQ \
  ./internal/pipeline/...
```

## IBM MQ build (optional)

A separate `Dockerfile.ibmmq` produces a CGO-linked binary with the
`ibmmq` build tag enabled, linked against the IBM MQ Redistributable
Client shipped under `ibmmq_dist/`. **The bundled IBM client is
`linux/amd64` only**, so the build must target that platform (it runs
under QEMU emulation on Apple Silicon hosts and takes longer):

```sh
docker buildx build --platform linux/amd64 \
  --build-context simpleauth-sdk=../SimpleAuth/sdk/go \
  -f Dockerfile.ibmmq -t mqconnector:ibmmq .
```

The runtime image is debian-slim because IBM's shared libraries are
linked against glibc; alpine's musl will silently fail at load time. The
default `Dockerfile` and `docker-compose.yml` produce the CGO-free image
(RabbitMQ + Kafka only).
